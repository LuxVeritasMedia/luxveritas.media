# Arie Quickstart

## Purpose

This is the short operational guide for continuing the LuxVeritas.media website build in Codex on another machine.

Read this first. Then read the full handoff if deeper context is needed.

## Open These Files First

In this order:

1. `AGENTS.md`
2. `docs/arie-handoff-website-build.md`
3. `tools/build-static.mjs`
4. `TODO.md`
5. `docs/strategy-round2.md`

## Current Build Model

The current site is a static prototype generated by:

- `tools/build-static.mjs`

Main assets:

- `styles.css`
- `app.js`

Generated routes are written directly into `.html` files.

## Commands To Run

From the project root:

```bash
node tools/build-static.mjs
node tools/serve-preview.mjs
```

Preview locally:

```text
Use the Local URL printed by `node tools/serve-preview.mjs`.
```

If testing on phone on same Wi-Fi, use the host machine IP:

```text
http://YOUR-LAN-IP:PRINTED-PORT/index.html
```

## What To Understand Before Editing

### Public Layer

Public pages should:

- sell the Lux Veritas world
- build trust
- convert fans
- capture access requests

Public pages should **not** expose:

- backend stack
- ops systems
- finance
- internal strategy
- campaign logic
- rights administration mechanics
- data-room style material

### Gated Layer

These should stay gated, access-based, or `noindex`:

- portal
- investor
- creator
- licensing
- deeper Codex layers
- internal/admin views

## What To Edit First

Start in this order:

1. `tools/build-static.mjs`
2. rebuild the site
3. inspect the generated HTML output
4. verify routes in browser
5. update `TODO.md` if something materially changes

Do not hand-edit generated HTML first unless you are debugging output. The generator is the source of truth.

## Highest-Priority Areas

If continuing the build, focus here first:

1. legal review follow-through for Privacy and Terms
2. launch-day write rehearsal with `LUX_PILOT_WRITE_TESTS=1`
3. Functions deploy IAM repair for the GitHub manual workflow
4. approved external workflow target selection for Google Workspace, GHL, or CodexOps
5. future migration path to Next.js if approved
