# LuxVeritas.media Project Instructions

## Product Identity

LuxVeritas.media is the institutional front door for Lux Veritas: music, film, events, Codex, artist submissions, partner access, private portal structure, future commerce, and audience ownership.

Tone is quiet luxury: calm authority, strong boundaries, selective disclosure, long memory, no hype-feed behavior.

## Product Boundary

This repo is the external public website. Keep public LuxVeritas.media work here: public pages, SEO, submissions, events, press, contact, audience capture, and public portal entry points.

Do not build internal LuxFlow OS app modules, DAMON, BlackGPT, AI Gateway, private audit logs, internal workspace/project state, admin dashboards, or private Firebase/Auth workflows in this repo.

Internal LuxFlow work belongs here:

```text
/Users/frederickparent/Documents/Codex/LuxFlow-OS
```

Public LuxVeritas.media work belongs here:

```text
/Users/frederickparent/Documents/Codex/LuxVeritas-website
```

Slogan rules:

- `Escape to Truth`: home hero and select campaign landings only.
- `Light Without Illusion`: Codex, film, and serious credibility moments.
- `Not Noise. Signal.`: music, tech, product, and supporting sections.
- `Truth, In Right Order`: footer, ledger, and internal contexts only.

## Navigation And CTA

Primary nav must not exceed six items. Phase 1 nav is exactly:

- Home
- Music
- Film
- Events
- Codex
- About

Utilities such as Portal, Press, Contact, Submissions, Ledger, Legal, Store, Works, Brands, Investor, Membership, Insights, and Community stay in the footer, as hidden routes, or as Phase 2 architecture.

Public CTA discipline: emphasize only `Request Access` and `Submissions`. Press/contact exists as utility routing and must not compete.

## Target Production Stack

Approved production build should migrate to:

- Next.js App Router
- TypeScript
- Tailwind CSS
- Firebase App Hosting
- Firebase Admin / Firestore for user role, tier, and `artistId`
- Supabase Postgres for relational content, events, Codex entries, form logs, release data, campaign links, products, and ledgers
- Supabase Edge Functions or server route handlers for webhook relays
- GoHighLevel for CRM, funnels, pipelines, tags, and workflows
- Stripe + Printify scaffolded server-side only

## Security Rules

- Never commit secrets, `.env`, API keys, webhook URLs, service-role keys, private Firebase keys, Stripe secrets, or Printify tokens.
- Do not expose GHL webhook URLs in client code.
- Public forms require validation, honeypot, consent fields, UTM capture, server-side relay, and a rate-limit hook before production.
- Admin routes must verify role server-side. Never trust client role claims.

## Design Rules

Use obsidian, bone, emerald, deep indigo, and restrained gold. Favor generous spacing, typography-led hierarchy, mobile-first layouts, accessible contrast, and subtle fade/reveal motion. Avoid autoplay video, loud parallax, cluttered navs, and heavy effects.

## Acceptance Direction

Before launch, production must include metadata, sitemap, robots, legal review placeholders resolved, form delivery tests, mobile QA, accessibility pass, Firebase deployment docs, Supabase schema docs, and no committed secrets.
